Problem classifier

Break classification

The classifier turns visual confusion into a controlled diagnostic. The question is not “what do we like?” The question is “which owner is emitting the visible break?”

Class A Missing mount

HTML loaded, but bridge or runtime did not mount.

Class B Loaded, not injected

Runtime file loaded, but no runtime root appeared.

Class C Competing stage

Old shell and new runtime both produce visible chamber output.

Class D Visual containment

The owner is correct, but spacing, motion, or clipping needs adjustment.

Execution probes

Four questions before any file move

These probes prevent false fixes. A TNT should only be written after the active owner and break class are named.

Probe 1

Which file owns the visible stage?

Identify whether the visible output comes from HTML, bridge JS, runtime JS, or cache.

Probe 2

Is the mount present?

If the mount is absent, the shell owns the failure. If the mount exists, inspect bridge/runtime.

Probe 3

Is the runtime root present?

If the runtime root exists, the issue is visual/layout. If it does not, the issue is load/injection.

Probe 4

Is there more than one stage owner?

If two generations render at once, remove competition before adding new features.

Direct audit lanes

Lane-based inspection

The lanes keep the diagnostic sequence clean. Do not skip from a visible symptom to a design rewrite.

Lane A

Shell lane

Inspect `/products/index.html` for mount point, page copy, filters, and static fallback.

Lane B

Bridge lane

Inspect `/products/index.js` for boot receipt, runtime request, cache-bust, and error handling.

Lane C

Runtime lane

Inspect `/products/products_runtime.js` for injected root, animation ownership, and layout control.

Lane D

Visual lane

Inspect containment, spacing, labels, glass depth, motion speed, and mobile readability.
