Authority / Route Authority Law + Route Law

Authority Law means every page, chamber, product, planet, law, route, or claim must know what is allowed to speak for it before it tries to impress anyone. A beautiful page without authority is only decoration.

Route Law means every chamber must know where it lives, how a visitor enters it, where it can send the visitor, and how the visitor returns. A route without a home becomes a hallway with no building around it.

Authority Law requires every route object to declare its parent, owner function, construct class, admissible scope, and proof authority before it can make design, runtime, product, scientific, or governance claims.

Route Law requires every public page to maintain explicit inbound, outbound, and return-path logic. A route must not operate as an isolated endpoint unless it is intentionally defined as a terminal chamber.

Governs

  • Page identity
  • Parent route
  • Owner function
  • Entry and return routes
  • Allowed page claims

Prevents

  • Floating pages
  • Unowned claims
  • Route drift
  • Confusing navigation
  • False authority
Audit questionCan this page state where it lives, what owns it, what it governs, what it does not govern, where it sends the visitor, and how the visitor returns?

Proof / Receipt Proof Law + Receipt Law

Proof Law means the bridge does not accept confidence as evidence. A claim must be checkable before it becomes reliable. If something cannot be seen, measured, tested, repeated, or traced, it may still be an idea, but it is not yet proof.

Receipt Law means every major change must leave a record. A receipt tells the next person what changed, what stayed untouched, what route owns the change, and what still needs verification.

Proof Law places observable evidence, route behavior, gauge results, source receipts, and diagnostic outputs above unsupported interpretation. No visual improvement, status claim, or narrative framing outranks verifiable behavior.

Receipt Law requires each major TNT or route renewal to preserve trace metadata including contract name, route target, changed scope, preserved scope, dependency posture, runtime posture, visual-pass status, and verification state.

Governs

  • Claims and evidence
  • Gauges and diagnostics
  • Version receipts
  • Deployment records
  • Verification trails

Prevents

  • False completion
  • Unsupported confidence
  • Lost change history
  • Visual-pass overclaiming
  • Post-deployment confusion
Audit questionCan the claim be checked, and does the file leave enough receipt data for another audit to understand what changed?

Runtime / Animation Runtime Law + Animation Law

Runtime Law means motion must serve the page instead of taking over the page. Not every chamber needs heavy movement. Some pages should guide, some should prove, some should display, and some should simply let the visitor move clearly.

Animation Law means motion can show relationship, hierarchy, and energy, but it cannot replace structure. If the page only makes sense while moving, the animation has become a crutch.

Runtime Law constrains motion authority to the route class that actually requires it. Static law pages must not import heavy render chains, canvas engines, or unrelated runtime systems unless the route contract explicitly authorizes them.

Animation Law allows visual motion only when it preserves semantic hierarchy, route usability, accessibility, and reduced-motion fallback. Animation may express relationships but may not become the source of route truth.

Governs

  • Motion behavior
  • Animation scope
  • Runtime containment
  • Reduced-motion safety
  • Static readability

Prevents

  • Heavy runtime on static pages
  • Motion hiding weak hierarchy
  • Decorative effects replacing function
  • Mobile usability damage
  • Reduced-motion failure
Audit questionDoes the page remain usable, understandable, and structurally accurate if animation is disabled?

Generation / Visual Claim Generation Law + Visual Claim Law

Generation Law means a page must not pretend a future stage is already complete. A baseline is a starting standard, not the finish line. Each generation should say what it proves and what still remains open.

Visual Claim Law means beautiful does not automatically mean true. A diamond, planet, orbit, animation, or polished layout can help people understand the page, but visual polish is not the same as source truth, scientific proof, readiness, or final approval.

Generation Law requires generation labels to remain bounded to the actual state achieved. Baseline, readiness, visual quality, source completeness, runtime stability, and proof status must remain distinct fields.

Visual Claim Law prohibits treating rendered appearance, generated graphics, screenshots, animation, or aesthetic polish as proof of source truth. Visual-pass status must remain false unless explicitly accepted through the proper review path.

Governs

  • Baselines
  • Generation labels
  • Visual readiness
  • Source truth
  • Status claims

Prevents

  • Calling a baseline complete
  • Treating polish as proof
  • Overclaiming screenshots
  • False readiness claims
  • Confusing beauty with verification
Audit questionIs the page claiming only what the current source, route behavior, receipt, and visual review actually support?

Human / Interaction Human Boundary Law + Interaction Law

Human Boundary Law means the system can help organize, explain, prepare, and clarify, but it must not pretend to replace qualified authority where qualified authority is required. The bridge can support decisions without overstepping its role.

Interaction Law means people should be able to tell what can be clicked, touched, opened, dragged, read, or ignored. A visitor should not have to guess which objects are controls and which objects are decoration.

Human Boundary Law requires high-stakes areas including medical, legal, financial, safety, scientific, official, and professional claims to preserve appropriate verification boundaries and not impersonate credentialed authority.

Interaction Law requires interactive elements to be visually and structurally distinguishable from decorative elements. Links, buttons, route diamonds, toggles, controls, and anchors must communicate function before activation.

Governs

  • Safety boundaries
  • Public claims
  • Clickable objects
  • Decorative objects
  • High-stakes support

Prevents

  • Reckless authority
  • Confusing support with final advice
  • Fake controls
  • Hidden interaction paths
  • Unclear interface behavior
Audit questionCan a user tell what this page is allowed to help with, what still needs verification, and what objects they can interact with?

Access / Return Access Law + Return Law

Access Law means the law layer must remain reachable. A governing chamber that cannot be found cannot govern the experience. The visitor should be able to reach Laws from the public route system without hunting.

Return Law means every outward movement needs a way back. Exploration should not become abandonment. The bridge can expand only if the visitor can re-anchor after each move.

Access Law requires core public routes, governing pages, proof pages, branch pages, and major chambers to remain discoverable from the route system. Laws must not become an orphan route.

Return Law requires every outward route, section jump, branch entry, or deep content movement to preserve an explicit re-anchoring path. On this page, grouped law summaries return to the large law-category diamonds at #law-diamonds.

Governs

  • Public reachability
  • Route loops
  • Return anchors
  • Page-changing routes
  • Visitor reorientation

Prevents

  • Isolated law pages
  • One-way navigation
  • Lost context
  • Outbound routes without return
  • Long-page disorientation
Audit questionCan a visitor reach the law chamber, move through it, and return to the correct category or route anchor without confusion?

Product / Foreground Product and Planet Separation Law + Foreground Law

Product and Planet Separation Law means different kinds of things must not be blended just because they share a visual style. Products, planets, laws, estate routes, proof routes, and story chambers each have different responsibilities.

Foreground Law means the most important controls must stay visible: law, entry, proof, and return. A beautiful page still fails if the visitor loses the governing controls.

Product and Planet Separation Law preserves route-class boundaries between product chambers, estate/navigation chambers, law chambers, scientific/proof branches, planetary constructs, and display/showroom objects.

Foreground Law requires governing controls such as Laws, Entry, Proof, Return, branch routes, and core navigation anchors to remain visible enough to preserve page function and audit usability.

Governs

  • Route class separation
  • Product identity
  • Planet identity
  • Law identity
  • Foreground controls

Prevents

  • Products pretending to be planets
  • Planets pretending to be products
  • Law routes becoming product pages
  • Proof routes buried under decoration
  • Important controls disappearing
Audit questionIs each object still in the correct route class, and are the governing controls still foregrounded?

Scope / Stale Language Scope Law + Stale Language Law

Scope Law means a change must stay inside the boundary that was authorized. If one file is being changed, the change should not silently become a full system rewrite. Scope keeps momentum from becoming damage.

Stale Language Law means old labels must not keep controlling current pages after the page has moved on. Old names, retired modes, and previous route logic can stay only when they are intentionally preserved.

Scope Law requires touched files to be delivered as full-file TNT replacements and confines the renewal to the authorized file, route, and contract. No partial snippets or unauthorized neighboring file mutations control project state.

Stale Language Law prohibits retired labels, old route identities, deprecated mode names, or prior generation language from remaining active unless the current route contract explicitly preserves them.

Governs

  • TNT scope
  • File boundaries
  • Route language
  • Retired labels
  • Current page identity

Prevents

  • Partial patches controlling state
  • Accidental rewrites
  • Old labels returning as active logic
  • Deprecated modes confusing visitors
  • Prior contracts overriding current bindings
Audit questionIs the change confined to the authorized file and free of old language that no longer controls the page?
