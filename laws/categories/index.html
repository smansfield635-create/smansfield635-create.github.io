Why categories matter
The Categories page turns law into a readable map. The Categories page is the interpretive map for /laws/, not a duplicate of /laws/.

Most systems fail because they treat every problem as the same kind of problem. A broken route, an unsupported claim, an overloaded runtime, a visual overclaim, and a missing return path are not the same failure. Categories let the visitor understand what kind of problem is being governed before the system tries to solve it.

This page preserves the eight grouped law categories and sixteen-law pairings while exposing the implementation logic through local category toggles. It maps category identity, paired laws, route behavior, color treatment, same-page anchors, and return controls without becoming the parent Laws chamber.

Expanded category definitions

Eight categories. Sixteen laws. One return field.

Each category answers a different question. Enter the public layer first, then switch to Engineering when the implementation boundary matters.

Each category card owns an independent CSS-only radio group. Local toggles affect only the local definition block and do not mutate route, orbit, return, receipt, or color state.

Authority / Route Authority Law + Route Law

Authority asks: who is allowed to speak here? Route asks: where does this thing live, where does it send the visitor, and how does the visitor return?

Without authority, a page can claim more than it owns. Without route, a visitor can enter a beautiful room and still get lost. This category keeps the system from becoming a collection of floating pages.

Authority / Route governs ownership, route identity, parent-child relationship, admissible scope, outbound movement, inbound movement, and return-path coherence.

A page must declare what it owns before it speaks, and a route must remain connected to the larger route system.

Governs

  • Route ownership
  • Parent identity
  • Admissible scope
  • Inbound movement
  • Return coherence

Prevents

  • Floating pages
  • False authority
  • Unowned claims
  • Route drift
  • Lost visitors
Category questionDoes this page know what it owns, where it lives, where it sends the visitor, and how the visitor returns?

Proof / Receipt Proof Law + Receipt Law

Proof asks: can this be checked? Receipt asks: can we tell what changed?

Confidence is not enough. A claim may sound right, look right, or feel right, but the system cannot treat it as reliable until it can be checked. A receipt keeps the change from disappearing into memory or opinion.

Proof / Receipt governs evidence hierarchy, route behavior, auditability, version markers, changed scope, preserved scope, deployment posture, and verification state.

Proof outranks confidence. Receipts preserve traceability.

Governs

  • Evidence hierarchy
  • Auditability
  • Changed scope
  • Preserved scope
  • Verification state

Prevents

  • Unsupported confidence
  • Missing receipts
  • False completion
  • Invisible changes
  • Opinion replacing proof
Category questionCan the claim be checked, and can the next room tell what changed?

Runtime / Animation Runtime Law + Animation Law

Runtime asks: does this page need motion? Animation asks: does motion clarify the structure, or is it hiding weak structure?

Motion can help the visitor understand relationship and energy. But motion should never become a substitute for meaning. A page must still make sense when the animation stops.

Runtime / Animation governs whether a page may load motion, whether that motion is lightweight, whether reduced-motion users are protected, and whether static readability survives.

Static law pages must not import heavy runtime unless explicitly authorized.

Governs

  • Motion authority
  • Animation purpose
  • Runtime limits
  • Static readability
  • Reduced-motion support

Prevents

  • Runtime overreach
  • Motion hiding weak structure
  • Heavy dependencies
  • Unreadable animation
  • Decorative confusion
Category questionDoes the page remain understandable when motion stops?

Generation / Visual Claim Generation Law + Visual Claim Law

Generation asks: what stage is this really in? Visual Claim asks: does the appearance prove anything, or only express something?

A polished page can still be unfinished. A beautiful diamond, planet, orbit, or route field can help the visitor understand the system, but beauty is not proof. The page must say what is real, what is staged, and what still needs verification.

Generation / Visual Claim governs baseline labels, readiness claims, visual-pass boundaries, staged development, source truth, and proof separation.

Visual polish cannot become evidence. A visual-pass claim remains forbidden unless explicitly reviewed and accepted.

Governs

  • Generation labels
  • Readiness claims
  • Visual boundaries
  • Source truth
  • Staged status

Prevents

  • Beauty becoming proof
  • False readiness
  • Visual-pass overclaiming
  • Baseline confusion
  • Evidence collapse
Category questionIs the page claiming only what the current source, route behavior, receipt, and review actually support?

Human / Interaction Human Boundary Law + Interaction Law

Human Boundary asks: where does the system stop? Interaction asks: can the visitor tell what they can do?

A system can guide, organize, teach, and clarify without pretending to replace human authority where human authority is required. A visitor should also be able to tell which objects are controls and which objects are decoration.

Human / Interaction governs high-stakes boundary language, role limitation, clickable controls, decorative objects, focus states, touch targets, label clarity, and interface affordance.

Interactive objects must communicate function before activation.

Governs

  • Human boundary
  • Role limitation
  • Clickable controls
  • Decorative objects
  • Focus clarity

Prevents

  • Authority impersonation
  • Fake controls
  • Hidden interaction
  • Unsafe certainty
  • Confusing interface objects
Category questionCan a visitor tell what the system can help with, what needs verification, and what can be clicked?

Access / Return Access Law + Return Law

Access asks: can the visitor reach the law? Return asks: can they get back after they move?

A law chamber that cannot be found cannot govern. A route that sends the visitor outward without a return path turns exploration into abandonment. This category keeps movement from becoming disorientation.

Access / Return governs discoverability, branch movement, section anchors, route loops, same-page return controls, and re-orientation behavior.

On /laws/categories/, Return to Orbit must target #catalog-groups.

Governs

  • Reachability
  • Route loops
  • Branch movement
  • Section anchors
  • Return target

Prevents

  • One-way movement
  • Lost context
  • Orphan chambers
  • Wrong return targets
  • Top-orbit confusion
Category questionCan a visitor enter, read, move, and return to the correct category field without confusion?

Product / Foreground Product and Planet Separation Law + Foreground Law

Product and Planet Separation asks: are different kinds of things being kept distinct? Foreground asks: are the most important controls still visible?

A product, a planet, a law, a story route, and a proof route may share visual language, but they do not share the same job. This category keeps objects from being blended together just because they look related.

Product / Foreground governs route-class separation, object identity, product boundaries, planet boundaries, law boundaries, and visibility of governing controls.

The foreground must preserve law, proof, entry, return, and route clarity.

Governs

  • Route class
  • Object identity
  • Product boundary
  • Planet boundary
  • Foreground controls

Prevents

  • Product/planet collapse
  • Law/product confusion
  • Proof route burial
  • Control disappearance
  • Visual family overreach
Category questionAre different route classes distinct, and are the governing controls still visible?

Scope / Stale Language Scope Law + Stale Language Law

Scope asks: are we changing only what was authorized? Stale Language asks: is old language still controlling a page that has moved on?

A small renewal can become damage if it silently expands. Old labels can also pull a page backward if they remain active after the contract changes. This category protects the current page from both overreach and outdated meaning.

Scope / Stale Language governs file boundaries, TNT scope, retired labels, current route identity, prior-contract handling, and stale semantic cleanup.

A contract renewal must be bounded to the authorized file and must not let deprecated language regain authority.

Governs

  • File boundary
  • TNT scope
  • Route identity
  • Prior contract handling
  • Stale cleanup

Prevents

  • Silent expansion
  • Old labels returning
  • Partial-patch control
  • Unauthorized mutation
  • Semantic drift
Category questionIs the renewal bounded to the authorized file and free of retired language regaining control?

Reality category branch
Reality is what still holds after the claim is tested. /laws/categories/reality/ is the active Laws Category branch standard.

Reality is not the first impression, the confident statement, or the desired outcome. Reality is what survives evidence, contradiction, consequence, pressure, measurement, limits, and audit.

The Reality category gem-field chain is: Reality → Theory → Evidence → Measure → Limits → Audit. The Reality Category route remains distinct from Reality Source and remains law-affiliated when represented as a route diamond.

Size, color, and return law
The visitor should know whether a diamond reads, leaves, or returns. Large white diamonds are same-page content. Small white diamonds are law-affiliated outbound routes. Small route-color diamonds are non-law outbound routes.

Some diamonds move within the category map. Some diamonds leave the page and still belong to Laws. Some diamonds leave the Laws family entirely. The page uses size and color so the visitor can tell the difference before clicking.

Large same-page category diamonds target anchors inside this document. Small white route diamonds target law-affiliated outbound routes such as /laws/scientific-law/ and /laws/categories/reality/. Small route-color diamonds target normalized non-law outbound routes: /, /door/, /home/, /explore/, /products/, /showroom/, /explore/frontier/, and /gauges/.
